Diagram 0 — Master flow (every ticket goes through this)

The chokepoint that runs on every create path (portal, agent, admin, API, email-to-ticket, ticket splits, legacy CSV import). autoAssignTicket runs after AI classification, automations, and the default-group backfill, so all three can affect what it sees. Three layers of defence keep tickets out of the no-group queue: resolveTicketGroup() at INSERT, backfillTicketGroupFromDefault() after hooks, and the hourly cron in scripts/process-stale-tickets.php.

flowchart TD
    A[End user / agent / API submits ticket] --> B["resolveTicketGroup()
caller's pick → type's group →
default_group_id setting →
lowest-id existing group"]
    B --> C["INSERT INTO tickets
(group_id rarely NULL)"]
    C --> D[runPostTicketCreateHooks]
    D --> E["classifyTicketWithAI
writes ai_classifications row
(skipped if type confidential
or AI provider disabled)"]
    E --> F["runAutomations 'ticket_created'
may overwrite group_id, type_id,
assigned_to via admin rules"]
    F --> G["backfillTicketGroupFromDefault
last-ditch NULL-group sweep
routes to default_group_id"]
    G --> H{assigned_to
already set?}
    H -->|yes — automation assigned it| ZA[Done — manually targeted by automation]
    H -->|no| I{group_id null?}
    I -->|yes — only on a fresh install
with zero groups| ZB[Sits in 'no group' queue
until cron sweep / triage]
    I -->|no| J["autoAssignTicket
reads groups.assign_strategy"]
    J --> K{strategy?}
    K -->|manual| D1[See Diagram 1]
    K -->|round_robin| D2[See Diagram 2]
    K -->|load_based| D3[See Diagram 3]
    K -->|skill_based| D4[See Diagram 4]
    K -->|ai_skill_based| D5[See Diagram 5]
    K -->|first_available| D6[See Diagram 6]
Auto-assign only runs when group_id is set AND assigned_to is still null. Automations execute before auto-assign — a "set group" automation can redirect the ticket, and a "set assignee" automation can short-circuit auto-assign entirely. The "no group queue" outcome is genuinely rare in 2.23+ — see Diagram 8 for the full default-group safety net.

Diagram 1 — Manual (the default)

groups.assign_strategy = 'manual'. Ticket is created and parked in the group queue waiting for a human.

flowchart TD
    A["Jane submits 'Printer Issue'
type's default group = IT"] --> B["tickets row created
group_id=3, assigned_to=NULL"]
    B --> C["autoAssignTicket reads
assign_strategy = 'manual'"]
    C --> D[Returns null — no pick]
    D --> E[Ticket waits in IT queue]
    E --> F["An IT agent opens the queue,
clicks 'assign to me' (or admin assigns)"]

Outcome: No auto-assign. Whichever IT agent grabs it first owns it.

Diagram 2 — Round Robin

Cycles through the group's members in user_id order. Ignores skill, load, and online status.

flowchart TD
    A["Jane submits 'Printer Issue'
group_id ← 3 (IT)"] --> B["autoAssign: strategy = round_robin"]
    B --> C["Get group members ordered by user_id:
Alice 5 → Bob 7 → Carol 9 → Dave 12"]
    C --> D["Read groups.assign_last_user_id = 7 (Bob)"]
    D --> E["Find Bob's index in member list = 1"]
    E --> F["Pick member at index (1+1) mod 4 = 2
→ Carol (id 9)"]
    F --> G["UPDATE tickets SET assigned_to = 9
UPDATE groups SET assign_last_user_id = 9"]
    G --> H[Carol gets the ticket]
    H -.- I["Next ticket → Dave (12)
Then → Alice (5, wraps)
Then → Bob (7)"]

Outcome: Carol. Bob is least-loaded with the right skill, but round-robin doesn't care.

Diagram 3 — Load Based

Picks whichever group member has the fewest currently-open tickets (any status except resolved or closed).

flowchart TD
    A["Jane submits 'Printer Issue'
group_id ← 3 (IT)"] --> B["autoAssign: strategy = load_based"]
    B --> C["Get group members: 5, 7, 9, 12"]
    C --> D["SELECT id, COUNT of open tickets
per assigned_to
WHERE status NOT IN ('resolved','closed')
ORDER BY count ASC, id ASC LIMIT 1"]
    D --> E["Result counts:
Alice=3, Bob=1, Carol=5, Dave=2
Bob wins (lowest count)"]
    E --> F[UPDATE tickets SET assigned_to = 7]
    F --> G[Bob gets the ticket]
    G -.- H["Tie-break: if Alice and Bob both
had 1 open, Alice wins (lower user_id)"]

Outcome: Bob. Doesn't check whether he's online or has the right skill — only load.

Diagram 4 — Skill Based (uses ticket-type required skills)

Filters group members to those whose skills cover every skill required by the ticket type, then picks least-loaded among them.

flowchart TD
    A["Jane submits 'Printer Issue'
type_id=17, group_id ← 3 (IT)"] --> B["autoAssign: strategy = skill_based"]
    B --> C["Get group members: 5, 7, 9, 12"]
    C --> D["Read ticket_type_skill_map
WHERE ticket_type_id = 17
→ required = {Printers}"]
    D --> E{Any required
skills?}
    E -->|no — empty list| FB["Fall back to assign_fallback
(Diagram 7)"]
    E -->|yes| F["Filter members holding ALL required skills
via user_skill_map"]
    F --> G["Eligible:
Alice (Printers, Networking) ✓
Bob (Printers, Servers) ✓
Carol — no Printers ✗
Dave — no Printers ✗"]
    G --> H{Any eligible?}
    H -->|no| FB
    H -->|yes| I["Least-loaded among eligible:
Alice=3, Bob=1 → Bob wins"]
    I --> J[Bob gets the ticket]

Outcome: Bob — has Printers AND fewer open tickets than Alice.

If the ticket type has no required skills checked, the strategy returns null and the fallback runs. Don't leave required-skills empty unless you mean to delegate to fallback.

Diagram 5 — AI Skill Based

Same shape as Diagram 4, but the "required skills" come from an AI classifier reading the ticket's content instead of a checkbox on the ticket type.

flowchart TD
    A["Jane submits ticket
Subject: 'Front desk printer keeps jamming'
Description: 'Jams every print'
Type: 'IT Issue' (general, no required skills)"] --> B["INSERT ticket, group_id ← 3 (IT)"]
    B --> C[runPostTicketCreateHooks]
    C --> D{Type confidential?
OR AI disabled?}
    D -->|yes| FB[Skip classification → fallback Diagram 7]
    D -->|no| E["classifyTicketWithAI builds candidate list:
SELECT FROM agent_skills
WHERE group_id IS NULL OR group_id = 3"]
    E --> F["POST subject + description + candidates
to Claude / OpenAI"]
    F --> G["Persist ai_classifications row:
suggested_skill_ids = [Printers]
confidence = 0.92
sentiment = neutral"]
    G --> H["autoAssign: strategy = ai_skill_based"]
    H --> I{confidence ≥
ai_confidence_threshold
default 0.7?}
    I -->|no| FB
    I -->|yes| J["Use overridden_skill_ids if admin set them,
else suggested_skill_ids"]
    J --> K["Filter group members holding ALL inferred skills"]
    K --> L["Eligible: Alice ✓, Bob ✓"]
    L --> M{Any eligible?}
    M -->|no| FB
    M -->|yes| N["Least-loaded among eligible:
Alice=3, Bob=1 → Bob"]
    N --> O[Bob gets the ticket]

Outcome: Bob — same as the explicit skill_based flow, but you didn't have to maintain a "Printer Issue" ticket type with required skills checked. One generic "IT Issue" type is enough.

Diagram 6 — First Available

Filters group members to those currently online (presence updated within the last 120 seconds), then picks least-loaded among them.

flowchart TD
    A["Jane submits 'Printer Issue'
group_id ← 3 (IT)"] --> B["autoAssign: strategy = first_available"]
    B --> C["Get group members: 5, 7, 9, 12"]
    C --> D["SELECT user_id FROM user_presence
WHERE user_id IN members
AND last_seen ≥ NOW() - 120 SECOND"]
    D --> E["Online subset:
Alice (last_seen 30s) ✓
Bob (last_seen 10m) ✗
Carol (last_seen 60s) ✓
Dave (no presence row) ✗"]
    E --> F{Any online?}
    F -->|no| FB[Fall back Diagram 7]
    F -->|yes| G["Least-loaded among online:
Alice=3, Carol=5 → Alice wins"]
    G --> H[Alice gets the ticket]

Outcome: Alice. Bob would have won on load alone but he's offline. Skills are not checked.

120-second window is generous on purpose — browsers throttle background-tab timers to ~once/minute, so 60s would mistakenly mark backgrounded agents as offline.

Diagram 7 — Fallback flow (when the primary strategy returns null)

Every non-manual strategy can return null (no eligible specialist, no one online, AI confidence too low, etc.). When that happens, autoAssignTicket reads groups.assign_fallback and runs that on the full group membership.

flowchart TD
    A["Primary strategy returned null
(no eligible candidate)"] --> B["Read groups.assign_fallback"]
    B --> C{fallback?}
    C -->|round_robin| D["Run round-robin on full group
(Diagram 2 logic)"]
    C -->|load_based| E["Run least-loaded on full group
(Diagram 3 logic)"]
    C -->|none| F["Return null
ticket waits in group queue
for human triage"]
    D --> G[Agent picked → assigned]
    E --> G
    F --> H[No assignment]

Common scenarios that trigger the fallback:

StrategyCondition that triggers fallback
skill_basedType has no required skills, OR no member has all of them
ai_skill_basedAI disabled, type confidential, AI failure, confidence below threshold, no member matches inferred skills
first_availableNobody in the group has been seen in the last 120 seconds
round_robin(rare) assign_last_user_id lookup fails — picks first member, no fallback needed
load_based(almost never) only fires if member list is empty

Diagram 8 — Default-group fallback (the "no ticket gets stuck" safety net)

A separate, earlier safety net than Diagram 7. Diagram 7 handles "no eligible agent in this group" — Diagram 8 handles "no group at all." Configured at Admin → Settings → Ticket Routing Defaults → Default Group. Three layers fire in sequence:

  1. At creation — every creation path calls resolveTicketGroup(), which chains caller's pick → ticket type's default group → default_group_id setting → lowest-id existing group.
  2. After post-create hooks — backfillTicketGroupFromDefault() sweeps any ticket still sitting with NULL group after AI classification and automations.
  3. Hourly cron — scripts/process-stale-tickets.php sweeps any orphans that the previous two layers somehow missed (legacy data, hand-edited rows, future creation paths a developer forgets to plumb).
flowchart TD
    A[Ticket creation path begins] --> B{Caller specified
a group?}
    B -->|yes| Z[Use that group]
    B -->|no| C{Type has a
default group?}
    C -->|yes| Z
    C -->|no| D{default_group_id
setting configured?}
    D -->|yes — and group exists| Z
    D -->|no or stale| E{Any groups
defined at all?}
    E -->|yes| F[Use the lowest-id
existing group]
    E -->|no| G["Group stays NULL
(only on a pristine fresh install)"]
    F --> Z
    Z[Ticket gets a group] --> H[Continue to Diagram 0]
    G --> I["Backfill happens later
in runPostTicketCreateHooks
and the hourly cron sweep"]
When this fallback fires it leaves an internal timeline entry on the ticket: "No group was matched by ticket type, AI, or automations — routed to the system default group so the ticket does not sit in the no-group queue." Triage spots unrouted arrivals via that note.
